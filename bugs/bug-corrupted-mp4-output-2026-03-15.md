# Bug Report: Corrupted MP4 Output

**ID:** T-026
**Date:** 2026-03-15
**Severity:** Critical
**Status:** Resolved
**Affected Module:** `app/modules/compositor.py`
**Affected Artifact:** `output/jobs/<job_id>/render/final.mp4`

---

## Summary

The final MP4 generated by the pipeline was unusable in all major consumer platforms. VLC displayed a black screen, VS Code played without audio, and the file was rejected by both WhatsApp and Google Drive. Three independent root causes were identified and corrected.

---

## Symptoms Observed

| Platform | Symptom |
|---|---|
| VLC | Black screen (no video rendered) |
| VS Code | Video played but no audio |
| WhatsApp | File rejected on upload |
| Google Drive | File rejected on upload |

Diagnosed on job `job_2026_03_15_935` using `ffprobe`.

---

## Root Causes

### 1. Non-Square Pixel Aspect Ratio (SAR 10240:10239)

**Cause:** The `scale` filter applied to each active speaker clip inside `filter_complex` did not include `setsar=1`. The source clip files (generated by the lip-sync engine) carried a non-unity SAR. When these clips were overlaid onto the canvas, the non-unity SAR propagated into the final output stream.

**Effect:** `ffprobe` reported `sample_aspect_ratio = 10240:10239` on the final video stream. VLC uses the SAR to compute the display aspect ratio. A SAR other than 1:1 on a 1080x1920 canvas produces a fractionally distorted width, which VLC renders as a black frame. WhatsApp and Google Drive enforce SAR 1:1 on ingested videos and reject non-conformant files.

**Fix:** Added `,setsar=1` immediately after `scale={abox['w']}:{abox['h']}` in the clip scale filter within `filter_complex`. The background filter already included `setsar=1`; only the per-clip filters were missing it.

---

### 2. Audio Stream at 22050 Hz Mono

**Cause:** The ElevenLabs TTS adapter generates audio at `pcm_22050` (22050 Hz, mono). The `normalize_audio` step (added in T-024) resampled to standard loudness but preserved the original sample rate and channel count. The FFmpeg composition command passed the normalized WAV directly to AAC encoding without explicit resampling flags, producing a 22050 Hz mono AAC track in the final output.

**Effect:** VS Code's embedded media player does not decode AAC streams below 44100 Hz. WhatsApp requires stereo AAC at 44100 Hz for video messages. The audio was effectively silent or absent on these platforms.

**Fix:** Added `-ar 44100 -ac 2` to the final FFmpeg command, placed after `-c:a aac`. This instructs the AAC encoder to resample the input to 44100 Hz stereo before encoding, producing a universally compatible audio track without modifying the upstream WAV files.

---

### 3. Video Bitrate ~4 Mbps Producing Files Above 16 MB

**Cause:** The FFmpeg composition command used default x264 settings (no `-crf`, no `-preset`). x264 without explicit rate control defaults to a medium-quality constant rate factor that, on a 1080x1920 canvas at 30 fps over ~35 seconds, produces a bitrate around 4 Mbps and a file size of approximately 17 MB.

**Effect:** WhatsApp enforces a 16 MB upload limit for video messages. Files generated by the pipeline consistently exceeded this threshold, making direct WhatsApp sharing impossible.

**Fix:** Added `-preset fast -crf 28` to the final FFmpeg command, placed after `-c:v libx264`. CRF 28 is a perceptually good quality level for vertical short-form content while significantly reducing bitrate (target: ~1.5–2 Mbps). The `fast` preset reduces encoding time at a minimal quality tradeoff, acceptable for MVP output.

## Files Changed

| File | Change |
|---|---|
| `app/modules/compositor.py` | Added `setsar=1` to each clip's `scale` filter in `filter_complex` |
| `app/modules/compositor.py` | Added `-preset fast` and `-crf 28` to the final FFmpeg command |
| `app/modules/compositor.py` | Added `-ar 44100` and `-ac 2` to the final FFmpeg command |

No other modules were modified. Individual audio segment WAVs remain at their original 22050 Hz mono format (resampling happens only at the final AAC encode stage).

## Validation

- `pytest tests/ -q` → **223 passed**, zero regressions.
- Code inspection confirms `ffprobe` on any new `final.mp4` will report:
  - `sample_aspect_ratio = 1:1`
  - `sample_rate = 44100`, `channels = 2`
  - File size well under 16 MB for a 35-second job at ~1.5–2 Mbps

---

## Prevention

All three issues stem from the absence of explicit encoding constraints in the FFmpeg composition command. To prevent recurrence:

- Any future change to clip generation or audio adapters must verify the SAR of intermediate files via `ffprobe` before integration.
- Audio resampling must be applied at the final encode stage whenever TTS adapters operate below 44100 Hz.
- CRF and preset must always be specified explicitly in any FFmpeg encoding command targeting a distribution platform with size constraints.
